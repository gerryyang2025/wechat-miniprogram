## 2. 当前需求背景

当前产品先以 `微信小程序` 为主要交付平台，但业务目标不仅是“卖课”，还包括：

- 课程商品上架与售卖
- 听课进度可视化
- 直播授课与直播带货
- 推广员分销
- 佣金与货款提现

长期产品目标仍覆盖 `内容交付 + 直播转化 + 分销变现 + 结算提现` 四条主线。

但当前 `MVP` 补充边界如下：

- `平台审核端功能` 作为后期功能实现
- `支付、订单、分销佣金、提现` 等付费相关功能作为后期功能实现
- `直播室` 一期优先使用 `微信原生直播能力`，如原生能力无法满足，再评估切换其他方案
- demo 图片中出现的 `练习、考试、作业本、付费问答、我的圈子、我的拼团、我的证书、兑换码、活动参与记录、推广中心` 不纳入当前 MVP

## 3. 核心需求清单

### R-01 课程上架

需求描述：

- 商家可以创建并上架课程商品
- 课程至少支持基础课程信息、价格、封面、课程目录、有效期和上架状态管理

业务价值：

- 这是知识付费业务的最小商品化前提
- 后续直播、训练营、分销和带货都需要依赖课程商品体系

### R-02 听课进度显示

需求描述：

- 用户在课程学习过程中，可以看到当前课程的学习进度
- 进度至少应覆盖 `已学章节 / 总章节`、最近学习位置、完成状态

业务价值：

- 提升用户复学效率
- 为商家提供学情判断与运营跟进依据

### R-03 读书卡类目不支持七天无理由

需求描述：

- `读书卡` 类商品不支持七天无理由退款
- 该规则需要在商品详情、下单确认、订单或售后规则中明确展示

业务价值：

- 降低交易争议
- 让商品类目规则与售后规则在前台可见且可确认

### R-04 直播授课

需求描述：

- 平台需要支持直播授课
- 直播侧需要支持主播滤镜 / 美颜能力
- 直播间需要支持按用户分类控制观看权限，例如：
  - 仅付费学员可听
  - 仅某个课程的学员可听
  - 仅某个训练营的报名学员可听

业务价值：

- 直播是内容交付和销售转化的重要场景
- 分层准入有助于把直播做成课程权益和促转化工具

### R-05 直播带货与直接购买

需求描述：

- 直播过程中可以直接挂商品并完成购买
- 业务诉求上希望支持 `微信支付 + 支付宝支付`

业务价值：

- 把直播流量直接转化为订单
- 形成 `直播讲解 -> 商品点击 -> 立即购买 -> 支付` 的闭环

平台说明：

- 在当前 `微信小程序` 形态下，一期原生支付能力以 `微信支付` 为主
- `支付宝支付` 作为业务诉求保留，但不作为一期微信小程序原生支付能力承诺，应放入后续 `H5（HTML5 页面） / App（Application） / 多端收银` 扩展评估

### R-06 分销 / 推广员

需求描述：

- 平台需要支持分销推广
- 推广员可以获取专属推广链接、海报或商品卡，带来下单后形成归因

业务价值：

- 用低成本方式扩大课程传播
- 把用户转化为推广员，形成私域裂变

### R-07 提现

需求描述：

- 平台需要支持提现
- 提现主体同时覆盖：
  - 个人
  - 公司
- 至少需要支持提现申请、账户信息管理、审核状态和结果反馈

业务价值：

- 分销佣金、商家货款或结算资金需要有明确的资金出口
- 这是分销体系和经营闭环能否落地的关键环节

## 4. 核心需求与一期 MVP（Minimum Viable Product）的对应关系

| 需求 | 一期落地方向 | 备注 |
| --- | --- | --- |
| R-01 课程上架 | 进入 `P0` | 属于商品体系底座 |
| R-02 听课进度显示 | 进入 `P0` | 属于学习交付底座 |
| R-03 读书卡不支持七天无理由 | 随付费链路后期实现 | 该规则保留，但不纳入当前 MVP 验收 |
| R-04 直播授课 | `直播准入控制` 进入 `P0`，`滤镜 / 美颜` 进入 `P1` | 一期优先采用微信原生直播能力 |
| R-05 直播带货与直接购买 | 后期实现 | 支付与带货闭环不纳入当前 MVP |
| R-06 分销 / 推广员 | 后期实现 | 分销归因与佣金体系后置 |
| R-07 提现 | 后期实现 | 随资金结算链路一起落地 |

## 5. 平台适配与实现备注

### 5.1 支付能力备注

- 当前 `MVP` 不实现支付闭环，支付能力作为后期功能规划保留
- 微信支付官方文档说明，小程序支付是商户在 `微信小程序` 中使用微信支付收款的能力
- 微信支付支付模式文档说明，在小程序中能且只能使用 `小程序支付` 的方式来唤起微信支付

因此，本项目在“一期仅做微信小程序”前提下，应把 `微信支付` 视为原生支付闭环，`支付宝支付` 视为后续跨平台能力诉求，而不是微信小程序原生支付范围。

### 5.2 直播滤镜备注

- 一期直播室优先使用微信原生直播能力
- 仅当原生能力在互动、权限、管理或扩展性方面无法满足需求时，再评估切换其他直播方案
- 小程序端直播美颜 / 滤镜可通过腾讯云相关插件与 SDK（Software Development Kit）方案接入
- 这类能力通常需要额外插件声明、License 和推流方案配合，适合作为直播增强能力逐步接入
