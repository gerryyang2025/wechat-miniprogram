## 2. 项目概述

`private-domain-operation` 目标是打造一款面向知识付费、社群运营、在线教育与内容营销场景的私域经营工具，帮助机构、讲师、品牌方和企业培训团队在微信生态内完成内容交付、用户转化、社群沉淀与持续变现。

一期产品以 `微信小程序` 作为唯一实现平台，优先打通微信生态中的关键经营路径：

- 内容上架与售卖
- 用户进入、购买、学习、留存
- 社群互动与训练营服务
- 营销裂变与分销传播
- 基础数据分析与运营复盘

后续再扩展到 `H5（HTML5 页面） / PC 管理台（Personal Computer Admin Console） / 独立 App（Application）` 等平台，形成统一的多端产品矩阵。

## 3. 产品定位

### 3.1 产品定义

一款以微信生态为核心阵地的 `私域知识产品经营平台`，提供：

- 知识内容生产与交付
- 课程、训练营、会员、电子资料等商品化能力
- 商城、分销、优惠、裂变等营销能力
- 学员、会员、社群、标签、学习档案等用户运营能力
- 订单、收入、转化、活跃、学习行为等经营分析能力

### 3.2 核心价值

对商家：

- 降低搭建知识付费和私域运营系统的技术门槛
- 在微信生态内完成从获客到成交、从学习到复购的闭环
- 沉淀用户数据与社群关系，而不是依赖公域平台

对学员：

- 在微信内低门槛完成购买、学习、打卡、互动
- 统一获得课程、直播、训练营、会员权益等服务
- 用更顺畅的路径参与内容学习和社群陪伴

## 5. 产品目标

### 5.1 一期目标

一期以微信小程序为核心，完成一个可用的私域内容交付与直播运营闭环，重点验证：

- 商家是否能快速创建并发布知识产品
- 学员是否能顺畅进入内容并完成学习
- 训练营与基础服务是否能支撑持续使用
- 微信原生直播是否能满足当前直播交付场景
- 运营数据是否足以支持日常经营决策

当前 `MVP` 补充边界：

- 平台审核端功能暂不进入一期实现
- 支付、订单、分销佣金、提现等付费相关能力暂不进入一期实现
- 直播室优先使用 `微信原生直播能力`，仅在原生能力无法满足场景时再评估切换方案
- demo 图中的独立练习、考试、作业本、付费问答、我的圈子、我的拼团、我的证书、兑换码、活动参与记录、推广中心等功能不进入当前 MVP
- 当前 MVP 先以核心需求相关的产品原型页面为主要交付物，直播、视频播放、学习进度等功能按模块分步骤接入

### 5.2 阶段性业务目标

- 支持商家快速开店与配置品牌主页
- 支持录播课、直播、训练营三种核心交付形态
- 支持基础内容运营与分享传播
- 支持学员沉淀、标签化管理与学习档案
- 支持微信内分享传播与订单转化追踪
- 支持从页面原型到模块化功能接入的渐进式实现路径

### 5.3 对标产品洞察

结合小鹅通官网公开信息，可以归纳出这类产品的成熟形态通常不是单一“课程工具”，而是由多个能力层叠加而成：

- `经营底座`
  - 私域营销
  - 电商平台
  - CRM（Customer Relationship Management） / SCRM（Social Customer Relationship Management）
  - 开放接入能力
- `内容交付`
  - 课程
  - 直播
  - 训练营
  - 圈子 / 社群
  - 企业培训
- `营销增长`
  - 优惠券、拼团、秒杀、分销、分享有礼等转化工具
- `用户运营`
  - 标签、分层、跟进、学习档案、社群沉淀
- `数据与智能`
  - 营收分析
  - 用户分析
  - 学情分析
  - AI（Artificial Intelligence）辅助运营

这意味着我们的产品设计不能只围绕“卖课”本身，而要从一开始就按 `商品交付 + 私域运营 + 用户沉淀 + 数据分析` 四层去规划。

详细的一期范围拆解见 [MVP_FEATURES.md](./MVP_FEATURES.md)。

## 6. 产品边界

### 6.1 一期纳入范围

- 微信小程序端
- 面向商家与学员的双角色产品能力
- 轻量商家工作台
- 面向学员的商城、学习、互动与会员服务
- 课程上架、学习进度、训练营、直播与基础数据分析

### 6.2 一期暂不纳入范围

- 独立 PC（Personal Computer）管理后台
- 独立 App（Application）
- 复杂财务对账与发票系统
- 高级自动化营销引擎
- 多组织、多租户集团级权限体系
- 深度 AI（Artificial Intelligence）内容生产与自动化客服
- 面向外部商家系统的 SDK（Software Development Kit）开放接入
- 平台审核端功能
- 支付、订单、分销佣金、提现等付费闭环能力

## 7. 产品形态方案

### 7.1 一期形态：单小程序双角色

一期建议采用 `单小程序双角色架构`：

- `用户端视角`：商城、课程学习、直播观看、训练营参与、会员服务、个人中心
- `商家端视角`：商品管理、订单管理、学员管理、营销活动、社群运营、数据看板

角色切换方式：

- 普通微信用户默认进入学员视角
- 具有商家后台权限的账号，可在个人中心切换到商家工作台

这样可以最大化利用微信生态优势，同时降低一期开发复杂度与冷启动成本。

### 7.2 后续扩展形态

二期及以后逐步扩展：

- `H5（HTML5 页面）`：用于分享承接、外部访问与活动页落地
- `PC 管理台（Personal Computer Admin Console）`：承接更复杂的内容管理、数据分析、配置管理、班主任运营
- `独立 App（Application）`：面向高频学习场景和品牌化客户
- `企业微信 / 公众号`：用于消息触达、私域导流和用户运营联动
- `开放接入 / SDK（Software Development Kit）`：面向已有官网、App、自有系统的商家，输出商品、订单、用户、培训能力

## 9. 核心业务场景

### 9.1 内容售卖场景

商家创建课程或训练营商品，用户进入小程序商城浏览内容详情，下单支付后立即获得学习资格。

### 9.2 直播公开课引流场景

商家通过微信社群、公众号或朋友圈分发直播链接，用户进入直播间完成观看、互动、关注和转化。

### 9.3 训练营陪跑场景

商家设计训练营计划，配置课程、打卡任务、作业与点评机制，班主任在小程序内管理学员进度与互动。

### 9.4 会员增长场景

商家提供会员专区、专属课程、电子书、优惠折扣和直播回放，用户购买会员后持续留存在私域。

### 9.5 老带新裂变场景

商家发起分销、拼团、邀请码、优惠券等活动，学员或推广员主动分享商品并带来新增用户与订单。

### 9.6 企业培训场景

企业以小程序为内部培训入口，员工通过微信或企业微信进入学习空间，参与图文课程、训练营任务、考试与排行榜，管理者查看部门培训进度和结果。

## 11. 技术与平台方案

### 11.1 一期平台策略

以微信小程序为核心平台，原因：

- 微信生态天然适合私域运营和社群传播
- 用户使用门槛低，购买与分享路径短
- 与公众号、企业微信、微信群联动成本低
- 适合快速验证内容付费与用户运营模型

### 11.2 技术文档拆分原则

为避免产品总设计文档过长，技术实现细节拆分为独立文档维护：

- [FRONTEND_TECH.md](./FRONTEND_TECH.md)
  - 微信小程序前端技术栈
  - 前端工程组织与实现约束
  - 性能与运行时适配原则
- [BACKEND_TECH.md](./BACKEND_TECH.md)
  - `Golang + Gin` 后端架构
  - `MySQL 8.0 + etcd + Linux SFTP + Nginx HTTPS` 存储与媒资分发分工
  - 接口、事务、模块和数据职责划分

### 11.3 技术方案摘要

- 前端：一期仅使用微信小程序原生技术栈
- 后端：一期使用 `Golang + Gin`
- 业务主数据：推荐使用 `MySQL 8.0`
- 轻量元数据与协调数据：使用 `etcd`
- 课程视频资源：优先使用 `Linux SFTP + Nginx HTTPS`，后续按规模评估 `COS / VOD`
- 直播室：一期优先使用 `微信原生直播能力`
- 消息与触达：优先使用微信生态原生能力

### 11.4 配套服务策略

参考对标产品的成熟做法，平台长期不应只交付软件，还应逐步补齐以下服务能力：

- 商家 onboarding 指导
- 场景化运营模板
- 培训与教练支持
- 大客户实施与定制对接
- 运维响应与问题追踪

### 11.5 二期扩展策略

扩展方向：

- `PC 管理台（Personal Computer Admin Console）`
  - 复杂商品管理
  - 深度数据分析
  - 批量运营配置
- `H5（HTML5 页面）`
  - 活动承接页
  - 外部流量落地页
- `App（Application）`
  - 高频学习
  - 离线缓存
  - 更重的学习体验
- `开放平台`
  - 向商家自有官网、App、CRM（Customer Relationship Management）或内训系统输出能力

扩展能力与开放平台规划，可结合 [13. 参考信息](#13-参考信息) 中的官网页面继续细化。

平台统一原则：

- 统一账号体系
- 统一商品体系
- 统一订单与权益体系
- 统一用户标签与学习档案
- 统一数据分析口径

## 12. 风险与难点

### 12.1 产品风险

- 一期如果功能铺得过大，容易导致交付过慢
- 商家端如果全部塞进小程序，复杂操作体验可能受限
- 内容交付和营销玩法需要平衡，避免产品过重
- 既要覆盖知识付费，又要承接企业培训，会放大信息架构复杂度

### 12.2 技术风险

- 小程序端复杂后台交互的性能与体验控制
- 直播互动与训练营高频操作的稳定性
- 微信原生直播能力与目标场景之间的适配边界
- 分销归因与订单状态一致性
- 多角色权限控制复杂度

### 12.3 业务风险

- 不同行业客户需求差异大
- 营销模块容易膨胀
- 训练营场景对运营服务要求高
