## 2. 一期前端定位

一期前端以 `微信小程序` 为唯一实现平台，优先服务以下目标：

- 支撑内容展示、学习、直播、训练营与基础运营主流程
- 快速打通微信生态内的登录、分享、消息触达与原生直播能力
- 在原生小程序运行时下保证稳定性、性能与发布效率

## 3. 技术栈要求

- 使用 `微信小程序原生技术栈`
- 页面层：`WXML`
- 样式层：`WXSS`
- 逻辑层：`JavaScript / TypeScript`
- 组件体系：`小程序原生组件 + 自定义组件`
- 平台能力：优先使用 `微信官方 API（Application Programming Interface）`

## 4. 实现约束

### 4.1 框架约束

- 一期前端只面向 `微信小程序`
- 不引入以多端复用为目标的跨端框架作为主实现方案
- 页面、组件、路由、授权、分享、消息订阅、直播等能力优先遵循微信小程序官方规范

### 4.2 工程约束

- 项目结构按 `用户端 / 商家工作台 / 公共组件 / 公共服务` 划分
- 组件优先按业务域拆分，避免纯页面内堆砌大段逻辑
- 公共服务层至少拆分为：
  - API（Application Programming Interface）请求层
  - 登录与鉴权层
  - 内容与学习服务层
  - 直播接入与回放层
  - 文件上传与播放层
  - 埋点与日志层

### 4.3 状态管理约束

- 业务状态管理保持轻量
- 优先按页面、组件、领域模块拆分，避免过重的全局状态耦合
- 用户登录态、店铺态、直播间短状态等跨页面信息可做有限全局管理

## 5. 性能与体验要求

- 优先适配小程序运行时限制，包括包体积、首屏加载、分包、长列表渲染与弱网体验
- 首页、学习页、我的页应优先保证首屏渲染速度
- 视频播放页、直播页、课程详情页应优先保证滚动、切换与媒体加载稳定性
- 对列表页、课程目录页、训练营任务页、直播评论区等高频滚动区域，需尽量避免无边界 setData

## 6. 平台能力接入范围

一期前端需要重点接入以下微信能力：

- 微信授权登录
- 手机号获取或绑定
- 分享卡片与海报传播
- 订阅消息
- 文件上传与下载
- 音视频播放
- 微信原生直播相关能力

## 7. 直播前端专项说明

- 一期直播室优先使用微信原生直播能力
- 一期直播先保证基础观看、聊天、问答和观看权限控制
- 直播带货、支付挂载等能力放到后期随支付链路一并实现
- `滤镜 / 美颜` 作为增强能力，保留插件化接入空间
- 直播页面应与普通课程播放页做结构分离，避免媒体逻辑和互动逻辑强耦合

## 8. 页面实现建议

结合当前产品规划，一期前端优先实现顺序建议为：

1. 核心原型页骨架：首页 / 学习 / 我的 / 商品详情 / 课程播放 / 训练营详情 / 直播详情 / 直播间
2. 商家工作台原型页骨架：工作台首页 / 商品管理 / 直播管理 / 用户管理 / 内容运营
3. 路由、导航、静态数据和 Mock 数据联调
4. 视频播放、资料附件、学习进度等内容交付能力
5. 直播列表接入与直播详情 / 直播间的原生直播能力接入
6. 消息通知、咨询反馈、系统设置等基础服务页

## 9. 课程播放资源前端接入建议

课程播放页在 `MVP` 阶段建议优先采用“静态示例视频 + 本地 Mock 数据”的接入方式，先跑通页面与播放主路径，再逐步替换成真实后端接口。

建议约束：

- 课程视频不要打进小程序代码包
- 播放资源优先使用 `HTTPS` 可访问地址
- 一期先准备 `3 ~ 5` 条课程示例视频即可
- 每条视频至少配套：
  - `course_id`
  - `lesson_id`
  - `title`
  - `cover_url`
  - `video_url`
  - `duration_seconds`
  - `progress_seconds`
- 前端课程播放页应先兼容：
  - 录播课视频
  - 训练营课程视频
  - 直播回放视频

前端页面落地建议：

- 原型阶段先通过统一 `mock/` 目录维护课程与视频数据
- `商品详情页 -> 课程播放页` 跳转时仅传递 `courseId` 与 `lessonId`
- 课程播放页进入后按 `courseId / lessonId` 读取本地 Mock 数据
- 封面加载失败时使用默认占位图，视频地址缺失时展示“资源准备中”状态
- 学习进度先用本地 Mock 值驱动，后续再切换成真实接口返回

这样可以保证：

- 不依赖后端即可先完成课程播放原型与交互验证
- 后续替换真实接口时，页面结构和参数模型不需要大改
- 视频资源准备与前端编码可以并行推进

## 10. 后续扩展预留

- 二期如扩展 `H5（HTML5 页面） / PC（Personal Computer） / App（Application）`，建议在当期重新评估是否引入跨端技术方案
- 当前阶段不为“未来可能多端复用”提前牺牲微信小程序原生体验和开发效率

## 11. Nano Banana AI 工具使用建议

`Nano Banana AI` 当前用于 `MVP` 原型阶段的页面交互图辅助生成，不直接替代前端编码实现。

建议使用方式：

- 页面原型图生成前，先以 [IA_PAGES.md](./IA_PAGES.md) 确认页面结构和模块顺序
- 直接使用 [PROTOTYPE_PROMPTS.md](./PROTOTYPE_PROMPTS.md) 中对应页面的完整提示词
- 每次只生成一张页面图，避免一次性生成多页面拼贴
- 优先生成 `首页、商品详情页、课程播放页、直播详情页、直播间页` 等主路径页面
- 第一轮先验证信息布局和模块层级，第二轮再微调视觉细节

使用约束：

- 原型图只作为产品沟通和前端占位实现参考
- 最终页面结构、组件拆分、路由和状态实现仍以 `IA`、`MVP` 和前端工程方案为准
- 生成内容必须保持中文界面、微信小程序风格和统一示例数据
- 原型图生成时应优先遵循 `9:19.5` 竖屏比例、`390 x 844` 逻辑尺寸和微信小程序安全区预留
- 输出图需适合移动终端查看，确保标题、正文、按钮和列表信息在手机端清晰可读
- 若生成结果偏离页面目标，应优先补充模块数量、关键状态和主 CTA 后重新生成
