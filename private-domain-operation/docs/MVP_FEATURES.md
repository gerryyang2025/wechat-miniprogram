## 1. 文档目标

这份文档用于明确 `private-domain-operation` 一期 MVP 的功能范围，帮助产品、设计、研发在立项阶段统一以下内容：

- 什么是一期必须做的核心功能
- 什么可以作为一期增强项处理
- 什么明确不进入一期
- 一期上线后如何判断产品已经具备最小可用闭环

## 2. MVP（Minimum Viable Product）定义

一期 MVP 的目标不是一次性做成完整“小鹅通”，而是优先打通一个可落地、可验证、可经营的微信小程序内容交付与直播运营闭环。

### 2.1 当前 MVP 补充边界

- `平台审核端功能` 作为后期功能实现，不进入当前一期开发与验收
- `支付、订单、退款、分销佣金、提现` 等付费相关能力作为后期功能实现
- `直播室` 一期优先使用 `微信原生直播能力`，仅在原生方案无法满足场景时再评估切换
- demo 图片中提供的 `练习、考试、作业本、付费问答、我的圈子、我的拼团、我的证书、兑换码、活动参与记录、推广中心` 不进入当前 MVP

这个闭环至少需要满足：

1. 商家可以开店并上架知识商品
2. 用户可以浏览、进入并消费内容
3. 商家可以通过课程、训练营、直播与基础运营手段提升留存和转化
4. 商家可以查看核心经营数据

### 2.2 当前 MVP 实施策略

当前 MVP 优先采用 `原型页面先行、功能模块后接入` 的推进方式：

- 第一步先完成核心需求对应的产品原型页面、页面结构、信息层级和主路径跳转
- 第二步使用静态数据或 Mock 数据完成前端页面联调，验证信息架构和交互路径
- 第三步再按模块逐步接入 `视频播放`、`学习进度`、`训练营打卡`、`直播观看` 等具体能力
- 直播、视频播放等媒体能力优先实现最小可用版本，后续再增强互动和细节体验
- 只要原型页和主路径已经稳定，即可先进入设计评审与前端搭建，不必等待全部业务能力一次性完成

## 3. 一期上线判断标准

如果同时满足以下四个条件，可以认为一期 MVP 第一阶段达到原型交付标准：

- 核心需求相关页面已经完成原型设计，并覆盖主要信息模块
- 用户侧主路径已跑通 `首页 -> 商品详情 -> 课程播放 / 训练营详情 / 直播详情`
- 商家侧主路径已跑通 `工作台 -> 商品管理 / 直播管理 / 用户管理 / 内容运营`
- 页面已支持静态数据或 Mock 数据联调，可进入下一步模块化功能接入
- 微信小程序内已可直接浏览基本页面内容和产品课程示例

## 4. 功能优先级分层

### 4.1 P0（Priority 0）

必须进入一期，缺少则无法构成最小闭环。

### 4.2 P1（Priority 1）

建议一期完成，但若资源有限，可在一期尾声或小版本补齐。

### 4.3 Out（Out of Scope）

明确不进入一期，留给二期或以后。

## 5. 一期范围总览

### 5.1 一期覆盖角色

- 学员 / 普通用户
- 商家管理员
- 讲师 / 内容主理人
- 班主任 / 运营人员

### 5.2 一期覆盖平台

- 微信小程序

### 5.3 一期不覆盖平台

- 如果图片功能与当前“一期只做微信小程序”的平台策略冲突，则先记录能力意图，延后完整实现

### 5.5 核心需求补充原则

当前业务方已明确提出一组必须覆盖的核心诉求，详见 [CORE_REQUIREMENTS.md](./CORE_REQUIREMENTS.md)。当前 MVP 范围需要额外遵循以下原则：

- `课程上架`、`听课进度显示`、`直播准入控制` 应优先进入一期
- `直播室` 一期优先采用 `微信原生直播能力`
- `直播带货`、`支付`、`分销 / 推广员`、`提现` 统一后置，不纳入当前一期验收
- `练习 / 考试 / 作业本 / 付费问答 / 我的圈子 / 我的拼团 / 我的证书 / 兑换码 / 活动参与记录 / 推广中心` 等 demo 派生功能不纳入当前 MVP
- `读书卡类目不支持七天无理由` 作为业务规则保留，待支付链路实施时一并落地

如需查看高层产品定位与能力架构，仍以 [DESIGN.md](./DESIGN.md) 为准。
